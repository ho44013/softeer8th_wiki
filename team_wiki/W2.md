## 1.1 아이디어 선정

처음 30분 동안 모든 팀원이 생각나는 주제와 해당 주제에 필요한 데이터, 타깃 고객을 간단히 정리하며 아이디어를 도출하였다.

총 13개의 아이디어 중 1차로 5개의 아이디어를 필터링하였다. 이후 각 아이디어에 대해 문제 상황과 시나리오를 간단히 설계하고, 샘플 데이터를 수집하여 실제 분석 가능성을 확인하였다.

최종적으로 **‘차기 굿즈 컨셉 선정’** 아이디어를 선정하였다.

## 1.2 비즈니스 문제 정의

### 문제 상황

자사에는 여러 캐릭터가 존재하지만, 어떤 캐릭터를 어떤 방식으로 상품화해야 할지에 대한 명확한 기준이 부족하다. 단순히 내부 선호도나 기존 인기 이미지에 따라 캐릭터를 선정할 경우, 실제 고객 반응과 맞지 않는 굿즈가 제작될 위험이 있다.

특히 최근 몇 년간 특정 캐릭터를 활용한 시즌 한정 굿즈가 기대보다 낮은 완판율을 기록한 사례가 있었다. 이에 따라 신규 굿즈 기획 시 **“왜 이 캐릭터와 이 컨셉으로 상품을 제작해야 하는가”** 에 대한 데이터 기반 근거가 필요해졌다.

### 핵심 문제

본 프로젝트의 핵심 문제는 단순히 **“어떤 캐릭터를 메인으로 선정할 것인가”** 가 아니다.

보다 정확한 문제는 다음과 같다.

> 고객이 각 캐릭터를 어떤 이미지와 감정으로 인식하고 있으며, 그 인식이 어떤 굿즈 컨셉으로 연결될 수 있는가?

### 제약 조건

* 굿즈 리뷰는 카카오프렌즈의 **춘식이** 및 **라이언**이 등장하는 상품을 대상으로 수집한다.
* 각 상품은 리뷰가 많은 순으로 정렬한 뒤, 상위 10개 품목에 대해서만 리뷰를 추출한다.
* 실제 시장에서 메인 캐릭터는 판매량, 수익률, 브랜드 대표성, 소비자 인지도 등 다양한 정량·정성 데이터를 종합적으로 고려하여 선정된다.
* 다만 본 실습에서는 기업 내부의 상세 판매 데이터와 수익률 자료에 접근할 수 없었기 때문에, 공개적으로 확인 가능한 카카오 공식 쇼핑몰의 베스트 상품 정보를 대체 지표로 활용하였다.
* 이를 바탕으로 **춘식이**를 메인 캐릭터로 선정하였으며, 춘식이의 대표성을 비교·검토하기 위한 기준 캐릭터로 **라이언**을 함께 채택하였다.

## 1.3 비즈니스 가치 설정

### 긍정 반응 키워드 활용

긍정 반응에서 자주 등장하는 단어는 굿즈 컨셉 선정에 활용할 수 있다.

예를 들어 특정 캐릭터에 대해 **‘힐링’**, **‘귀여움’**, **‘선물’**, **‘소장’** 과 같은 단어가 반복적으로 등장한다면, 해당 캐릭터는 감성형 굿즈나 선물용 굿즈에 적합할 가능성이 있다.

### 부정 반응 키워드 활용

부정 반응에서 자주 등장하는 단어는 굿즈 제작 시 피해야 할 요소나 개선 포인트를 알려준다.

예를 들어 **‘비싸다’**, **‘작다’**, **‘마감’**, **‘불편하다’** 와 같은 단어가 반복적으로 등장한다면, 가격 정책, 크기, 품질, 사용성 측면에서 개선이 필요하다고 볼 수 있다.

### 마케팅 문구 및 상품 설명 활용

고객이 캐릭터를 인식하는 감정 키워드는 마케팅 문구와 상품 설명의 설득력을 높이는 데 활용할 수 있다.

예를 들어 고객들이 특정 캐릭터를 **‘힐링’** 으로 인식한다면, 상품 상세페이지와 광고 문구에서 **‘하루를 위로하는 캐릭터 굿즈’** 와 같은 방향을 사용할 수 있다.

## b. 데이터 변환

### 평점 기준 긍정/부정 판단의 한계

고객이 남긴 평점이라는 지표를 바탕으로 리뷰 텍스트의 긍정 및 부정을 판단할 수 있으나, 실제 데이터에서는 편향이 존재하는 것을 확인하였다.

#### 1. 평점-텍스트 불일치

별점이 높아도 리뷰 본문에는 부정적 표현이 섞여 있는 경우가 많다.

예를 들어 다음과 같은 리뷰가 있을 수 있다.

> 배송은 늦었지만 상품은 괜찮아요.

이 경우 평점은 4점으로 긍정 분류될 수 있지만, 리뷰 본문에는 **‘늦었다’** 와 같은 부정 표현이 포함되어 있다. 이 단어가 긍정 워드클라우드에 그대로 포함되면 실제 긍정 요인과 맞지 않는 결과가 만들어질 수 있다.

반대로 낮은 평점의 리뷰에서도 특정 부분만 칭찬하는 경우가 있어, 감성 라벨과 실제 단어 내용이 어긋나는 경우가 존재하였다.

#### 2. 평점 인플레이션

실제 만족도와 무관하게 만점을 주는 형식적 리뷰가 긍정 그룹에 과대 포함되는 문제도 있었다.

이 경우 워드클라우드가 진짜 만족 요인보다 **‘좋아요’**, **‘감사합니다’** 와 같은 상투어 중심으로 구성될 수 있다.

따라서 리뷰 문맥을 파악하는 도구를 통해 각 키워드를 기반으로 분류할 필요가 있다고 판단하였고, 형태소 분석기를 사용하게 되었다.

### Mecab 채택 이유

본 프로젝트에서는 **Mecab**을 채택하였다.

#### 대용량 데이터 처리에 유리한 이유

* **Double-Array 사전 구조**

  * 사전 크기가 커져도 조회 속도가 거의 `O(문자수)` 수준으로 유지된다.
  * 사전 규모 확장에 강하다.

* **CRF + Viterbi 최단경로 탐색**

  * 문장을 그래프로 놓고 동적 계획법으로 한 번에 최적 경로를 계산한다.
  * 규칙 기반 재귀 탐색보다 연산량이 적다.

* **C++ 네이티브 엔진**

  * 순수 Python 기반 분석기나 규칙 기반 분석기보다 반복 대량 호출 시 오버헤드가 적다.

* **사전 사전 컴파일**

  * Binary Dictionary를 사용하므로 대량의 문장을 스트리밍 또는 배치 방식으로 처리할 때 초기 로딩 이후 처리 속도가 일정하게 유지된다.

#### 속도가 빠른 이유

* Double-Array TRIE 방식을 사용한다.
* 트리 노드를 포인터로 따라가는 대신 배열 연산을 사용하므로 캐시 친화적이다.
* 메모리 접근이 예측 가능해 매우 빠르게 동작한다.

## 2.6 WordCloud 분석 결과

## a. 워드클라우드 분석을 통한 인사이트 및 상품 기획 방향

### 1. 신제품 기획: 차기 십이간지 에디션 출시

라이언의 긍정 키워드에서는 **‘푸른뱀 복주머니’**, **‘데코인형’** 등 십이간지 에디션과 관련된 키워드가 확인되었다. 반면 춘식이 에디션에서는 유사한 키워드가 두드러지게 나타나지 않았다.

따라서 라이언 십이간지 시리즈의 성공 사례를 벤치마킹하여, 차기 십이간지 에디션의 메인 모델로 **춘식이**를 적극 검토할 필요가 있다.

### 2. 캐릭터 차별점: 실용성 중심의 굿즈 확장성 확인

라이언과 춘식이의 긍정 키워드를 비교했을 때, 두 캐릭터가 소비자에게 인식되는 방식에 차이가 있었다.

| 캐릭터 | 주요 키워드             | 해석                |
| --- | ------------------ | ----------------- |
| 라이언 | 안전요정, 출산선물, 복주머니   | 의례적·선물용 맥락이 강함    |
| 춘식이 | 만족, 좋아함, 가습기, 수면바지 | 실생활 밀착형 카테고리와 연결됨 |

이를 통해 라이언 대비 춘식이는 **실용적 라이프스타일 굿즈** 확장에 강점을 지닌 캐릭터로 판단할 수 있다.

### 3. 카테고리 만족도: 촉감/재질 중심 상품과의 시너지

다양한 카테고리 중 특히 **‘베이비필로우’** 의 비중이 가장 높게 나타났다.

이는 촉감과 재질이 강조되는 상품군이 춘식이 캐릭터 아이덴티티와 높은 시너지를 낸다는 점을 보여준다.

## b. 주요 개선 및 추가 분석 요청 사항

### 1. 부정 키워드 원인 분석 필요

부정 키워드에서는 **‘가격’**, **‘약하다’**, **‘냄새’** 등의 단어가 등장하였다.

해당 키워드들이 전반적인 제품 컨셉 결정이나 핵심 방향성을 저해할 수준은 아니지만, 구체적인 원인 파악이 필요하다.

### 2. 냄새 및 품질 불만 이슈의 카테고리 세분화 필요

지난 분석에 이어 **‘냄새’**, **‘포장’**, **‘반품’** 등의 부정 키워드가 지속적으로 노출되었다. 다만 현재 데이터만으로는 해당 키워드가 어떤 상품군에서 주로 발생했는지 명확하게 구분하기 어렵다.

예를 들어 다음과 같이 해석이 달라질 수 있다.

| 카테고리   | 해석 가능성                                  |
| ------ | --------------------------------------- |
| 방향제    | “냄새가 약하다”는 의견은 향 지속력 등 품질 불만으로 해석될 수 있음 |
| 베이비필로우 | 제조 공정, 보관, 유통 과정상의 문제일 가능성이 있음          |

따라서 향후 분석에서는 부정 키워드를 상품 카테고리별로 세분화하여 원인을 파악할 필요가 있다.

---

## 2.7 기획 단계의 비즈니스 가치 재현 가능성

기획 단계에서 설정한 비즈니스 가치가 실제 분석 과정에서 어느 정도 재현 가능한지 확인하였다.

### 긍정 키워드 측면

긍정 키워드로 분류된 단어들을 통해 추후 제작할 굿즈의 컨셉을 어느 정도 고려해볼 수 있음을 확인하였다.

특히 비교군에서 인기가 좋았던 제품의 컨셉을 참고하여 신규 굿즈 기획 방향으로 활용할 수 있다는 점에서 의미 있는 가치가 도출되었다.

### 부정 키워드 측면

부정 키워드는 컨셉을 결정짓는 데에는 크게 도움이 되지 않았다.

부정 키워드의 대부분은 실제로 수령한 제품에 대한 불만을 표현하는 내용이었다. 따라서 부정 키워드는 굿즈 컨셉 선정보다는 제품 품질, 가격, 배송, 포장 등의 개선 포인트를 찾는 데 더 적합하다고 판단하였다.

# 3. 미션 6: Docker 이미지를 AWS EC2에 배포하기

## 3.1 Docker를 사용하는 이유

### 1. 개발 환경과 실행 환경을 동일하게 맞출 수 있음

Docker를 사용하는 가장 큰 이유는 개발 환경과 실행 환경을 동일하게 맞출 수 있다는 점이다.

로컬에서 정상적으로 돌아가는 프로그램이 다른 사용자의 로컬이나 서버에서는 실행되지 않는 경우가 많은데, 이러한 문제의 대부분은 환경 불일치에서 발생한다.

`venv`, `conda`, `uv`처럼 가상 환경을 만들어 개발 환경을 맞추는 도구도 있지만, Docker는 이와 달리 **실행 환경 전체를 이미지로 묶어서 배포한다**는 점에서 차이가 있다.

### 2. 여러 머신에 동일한 환경을 빠르게 배포할 수 있음

데이터 엔지니어링에서는 Scale-out 방식으로 여러 머신에 데이터를 분산 처리하는 경우가 많다.

이때 머신마다 환경 설정과 실행 환경을 일일이 맞추면 불필요하게 많은 시간이 소요된다. Docker를 사용하면 동일한 이미지를 기반으로 여러 머신에 같은 실행 환경을 빠르게 구성할 수 있다.

### 3. 버전 관리와 공유가 쉬움

Docker 이미지는 버전 관리가 가능하기 때문에 이전 버전으로 롤백하기 쉽다. 또한 Docker Hub와 같은 이미지 저장소를 통해 이미지를 공유하기도 쉽다.

공유받은 사용자는 각자의 기존 OS와 관계없이 프로젝트에서 요구하는 OS와 실행 환경 위에서 프로그램을 실행할 수 있다.

예를 들어 Frontend와 Backend를 분리 개발할 때 서로 다른 Java 버전을 요구하는 경우에도 Docker를 사용하면 프로젝트별 실행 환경을 독립적으로 구성할 수 있다.

또한 `docker compose`를 사용하면 개발 환경과 운영 환경을 분리하여 개발 및 테스트를 진행하는 것도 가능하다.

### 4. 실행 관리가 편리함

Docker를 사용하면 `nohup`이나 `gunicorn`만으로 직접 프로세스를 관리하는 것보다 실행 관리와 로그 확인이 수월하다.

Docker Desktop, Portainer 등의 도구를 활용하면 컨테이너 실행 상태, 로그, 중지, 재시작 등을 더 쉽게 관리할 수 있다.

또한 같은 시스템을 여러 개 띄울 때 컨테이너 내부에서는 동일한 포트 번호를 사용할 수 있다는 점도 편리하다.

예를 들어 컨테이너 내부에서는 모두 `5432`, `8888`과 같은 포트를 사용할 수 있다. 단, 호스트와 연결하는 매핑 포트는 서로 다르게 지정해야 한다.

---

## 3.2 Docker 사용 시 불편한 점

### 1. 이미지 크기 문제

Docker 이미지는 빌드 방식에 따라 크기가 크게 달라질 수 있다.

이미지 크기가 커지면 다음과 같은 문제가 발생할 수 있다.

* 빌드 시간이 길어진다.
* Pull 시간이 길어진다.
* 배포 시간이 길어진다.
* EC2와 같은 서버 자원에 부담을 줄 수 있다.
* 이미지 크기 때문에 실행이 어려워지는 경우도 있다.

### 2. 디버깅 관점이 늘어남

Docker를 사용하면 기존 코드 문제 외에도 다음과 같은 추가적인 관점을 함께 고려해야 한다.

* 코드 자체의 문제인지
* 의존성 문제인지
* Dockerfile 작성 문제인지
* 이미지 빌드 문제인지
* 컨테이너 실행 옵션 문제인지
* 포트 매핑 문제인지
* 볼륨 마운트 문제인지

즉, 문제가 발생했을 때 **“이것이 Docker의 문제인가?”** 라는 관점이 추가되어 디버깅해야 할 요소가 늘어난다는 단점이 있다.

## 3.3 여러 대의 EC2에 여러 컨테이너를 배포한다면

이번 미션에서는 하나의 EC2에 하나의 Docker 컨테이너를 배포하였다.

하지만 여러 대의 EC2에 여러 개의 컨테이너를 배포해야 한다면, 손으로 직접 배포하는 방식에는 한계가 있다. 컨테이너 하나를 배포하는 데에도 시간이 걸리는데, 이를 여러 대로 확장하면 수동 관리가 매우 어려워진다.

따라서 각 배포 과정을 자동화할 수 있는 오케스트레이션 플랫폼의 도움을 받아야 한다.

대표적인 도구는 다음과 같다.

* Kubernetes, k8s
* Amazon EKS
* Docker Swarm
* ECS

또한 여러 대의 EC2에 여러 컨테이너를 배포하는 경우, Dockerfile이나 Compose 파일을 더 엄격하게 관리해야 한다.

추가적으로 필요한 요소는 다음과 같다.

### 1. 컨테이너 배포 권한 및 체계 구축

여러 사용자가 여러 서버에 컨테이너를 배포하게 되면, 누가 어떤 컨테이너를 배포할 수 있는지에 대한 권한 관리가 필요하다.

이를 위해 다음과 같은 체계를 고려해야 한다.

* 배포 권한 관리
* 이미지 저장소 접근 권한 관리
* 환경 변수 및 Secret 관리
* 운영 환경과 개발 환경 분리

### 2. EC2 간 컨테이너 배치 및 Balancing

여러 EC2에 컨테이너를 분산 배포할 경우, 각 EC2가 수용할 수 있는 컨테이너 수와 자원을 고려해야 한다.

이를 위해 다음과 같은 요소를 관리해야 한다.

* CPU 사용량
* Memory 사용량
* 네트워크 트래픽
* 컨테이너별 우선순위
* 장애 발생 시 재배치
* 부하 분산

결과적으로 여러 대의 EC2에 여러 컨테이너를 배포하는 상황에서는 단순히 Docker만 사용하는 것이 아니라, 컨테이너 오케스트레이션 도구와 배포 자동화 체계를 함께 구성해야 한다.
