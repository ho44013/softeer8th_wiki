# W3M2b - Understanding of Hadoop Configuration Files
> 4개의 xml files들을 살펴 보고 각 화일의 셋팅 중에 중요하거나 유용하다고 생각되는 것들을 각자 골라서 용법을 파악해 보세요. 그런 다음 팀과 함께 토의한 다음, 위키에 정리합시다.

### core-site.xml
- fs.defaultFS : 시스템 기본 주소. 전반적인 작업을 관리하는 파일 시스템의 주소를 설정한다. 싱글노드 하둡 실습(M1)에서는 localhost로 두어도 작동했으나, 멀티노드 하둡에서는 이를 명시적인 이름(과제에서는 namenode)으로 지정해주어야 했다.
- hadoop.tmp.dir 임시 파일들을 저장하는 기본 뼈대. 만약 없으면 컨테니어를 껐다키면 데이터가 날아감

### hdfs-site.xml
- replication : 워커 개수 지정 - 워커 없으면 하둡 필요없음
- dfs.blocksize : 기본 블록 크기를 지정 - 워크로드에 따라 적절한 값을 설정하는 것이 중요
- dfs.namenode.name.dir : NameNode 메타데이터 저장 위치 - 파일 시스템의 메타데이터를 저장하는 위치로, NameNode 운영에 필수

### mapred-site.xml
- sort.mb : 정렬
- mapreduce.framework.name=yarn : MapReduce 작업을 YARN에서 실행하도록 지정 - 없으면 Local Mode로 실행되어 YARN 클러스터를 사용하는 의미가 없음
- mapreduce.application.classpath : MapReduce 실행 시 필요한 JAR 파일들의 경로 지정 - ApplicationMaster와 Task가 Hadoop 라이브러리를 찾지 못하면 ClassNotFoundException이 발생함.
- mapreduce.map.env, mapreduce.reduce.env, yarn.app.mapreduce.am.env : 각 Map, Reduce, ApplicationMaster에 HADOOP_MAPRED_HOME 환경변수 전달 — Docker에서는 컨테이너마다 환경변수가 독립적이므로 이를 명시하지 않으면 MapReduce 실행이 실패하는 경우가 많음

### yarn-site.xml
- yarn.nodemanager.resource.memory-mb + yarn.scheduler.maximum-allocation-mb : mapreduce 작업을 수행하는 데에 소모되는 메모리보다 이게 적으면 Job이 ACCEPTED 상태에서 멈춘다
- yarn.nodemanager.aux-services : 이거 shuffle로 설정 안 하면, MRjob 터짐


# W3M4 - Twitter Sentiment Analysis using MapReduce

> 'predefined keywords'가 아닌 다른 방법으로 나누고자 한다면 어떤 방법이 있을까요?

- SVM, Naive Bayes, Random Forest 등 ML model 학습시켜서 라벨링 하기
- LLM(GPT, Llama 등)을 활용한 프롬프트 기반 분류
- Vader 같은 룰 기반 Lexicon 이용하기 
- BERT 등 사전학습 언어모델 이용하기
- 단어별 가중치 점수


## Vader를 활용한 추가 진행

### 무엇을 했나
Sentiment140 학습셋에서 긍정/부정 각 500건씩 총 1000건을 샘플링한 뒤, 사전 정의 키워드 대신 VADER(규칙 기반 lexicon 감정분석기) 로 감정을 분류하고 하둡 MapReduce(Streaming)로 집계했다.

### VADER를 쓴 이유
VADER는 학습이 필요 없는 규칙 기반 분석기로, 단어별 감정 가중치에 더해 부정어("not good"), 강조어("very"), 문장부호("!!!"), 대문자, 이모지 같은 문맥 신호까지 반영한다. 단순 키워드 개수 세기보다 트윗 같은 SNS 텍스트에 잘 맞는다. 각 트윗의 compound 점수(−1~+1)로 ≥0.05 → positive, ≤−0.05 → negative, 그 사이는 neutral로 분류했다.

### 결과 (1000건)
positive 466, neutral 284, negative 250 — 합계 1000으로 누락 없이 집계됐다. 원래 소스 라벨은 긍정 500/부정 500이었는데, VADER 분류에서는 긍정이 466으로 비슷하게 잡힌 반면 부정 라벨 500건 중 상당수가 neutral(284)로 넘어갔다. 

### 의미
키워드 방식이 "매칭 없음/동점 → neutral"로 대부분을 neutral에 몰아넣던 것과 달리, VADER는 문맥 규칙으로 긍/부정을 더 적극적으로 판정해 세 범주에 고르게 분포시켰다. 다만 규칙 기반의 한계로 부정·비꼼 탐지는 여전히 약하며, 더 높은 정확도가 필요하면 소스 라벨(0/4)로 학습하는 지도학습 방식이 다음 단계가 된다.
